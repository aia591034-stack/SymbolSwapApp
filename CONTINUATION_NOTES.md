# プロジェクト継続ノート: symbol-swap-app

## 1. プロジェクト概要
Symbol SDK v3 を使用した、アトミックスワップ形式のファイル売買マーケットプレイス。
- **A (運営)**: 手数料を支払うアグリゲートトランザクションの署名者。
- **B (ユーザーA)**: 出品者（または購入者）。
- **C (ユーザーB)**: 購入者（または出品者）。

## 2. 現在のステータス
- **修正完了**: 
    - SDK v3 のバイナリ化エラー (`offset is out of bounds`) 回避。
    - 署名検証エラー (`Ineligible_Cosignatory`) 解消。
    - ネットワーク禁止エラー (`V1_Prohibited`) 解消。
    - 接続先を `https://sym-test-01.opening-line.jp:3001` に統一。
    - `index.html` 内での公開鍵指定バグ（非公開鍵を渡していた）を修正。
    - アカウントA, B, C の残高確認完了（いずれも十分な残高あり）。
    - **Symbol SDK v3 の読み込み修正**: サーバー側は v3、フロントエンド（SSS連携用）は v2 を使用するハイブリッド構成に変更。
- **アグリゲートトランザクション V2 対応 (2026-04-22)**:
    - `Failure_Aggregate_V1/V2_Prohibited` エラーを解消するため、トランザクションの構築をサーバー側（SDK v3）に移行。
    - `/api/build_transaction`: サーバーで正しい V2 形式（Strict Aggregate Hash）を生成し、運営の主署名を付与してペイロードを返す。
    - `/api/announce_transaction`: クライアント（SDK v2）で作成したコサイン署名をサーバー側で結合して送信する。
    - `index.html`: ブラウザでの構築を廃止し、サーバーから取得したペイロードに署名するフローへ変更。
- **バグ修正**: `build_transaction` 時の商品ID型不一致（404エラー）と、Deadline計算ミスを修正。

## 3. 次のステップ
1.  **署名結合の検証**: サーバー側で構築した V2 ペイロードに対し、スマホ（SDK v2）の署名が正しく結合され、アナウンスが成功するか確認。
2.  **アグリゲート・ボンデッド（Aggregate Bonded）への拡張**:
    - 現状は「コンプリート」形式。出品者が運営でも購入者でもない場合、出品者のコサイン署名が不足してエラーになる可能性がある。
    - 必要に応じて、ハッシュロックを伴うボンデッド形式への変更を検討。
3.  **SSS 連携の再確認**: サーバー構築フローに変更したため、PC版 SSS 拡張機能での動作も再確認が必要。
